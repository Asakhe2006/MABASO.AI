# MABASO AI

Live site: https://mabaso-ai-web.onrender.com/

Mabaso AI is a study workspace for lecture capture, transcripts, study guides, tests, collaboration, and a secure lecture conversation assistant.

## Stack

- Frontend: React + Vite + Tailwind-style utility classes
- Frontend runtime/build toolchain: Node.js 20.19+
- Backend: FastAPI
- Database: SQLite with a Render persistent disk in production
- Deployment: Render

## Security Model

- The React website is built with Node.js/Vite, while sensitive application logic stays on the backend.
- New login sessions use signed JWT access tokens generated by the backend.
- Session tokens are set in Secure, HttpOnly cookies so browser JavaScript does not need to persist the secret in localStorage.
- Cookie-authenticated write requests use a signed CSRF cookie plus `X-CSRF-Token` header.
- The backend still accepts older bearer tokens during migration, but new frontend sessions use cookie-backed auth.
- Server-side session hashes remain in the database, so logout, admin force logout, account suspension, and session revocation can take effect immediately.

## Lecture Assistant

The lecture assistant now uses secure backend streaming and never exposes provider API keys in the browser.

Flow:

`Browser UI -> Mabaso backend -> Gemini / Groq / OpenRouter -> streamed reply`

Endpoints:

- `POST /api/chat/stream`
- `POST /api/chat/gemini`
- `POST /api/chat/groq`
- `POST /api/chat/openrouter`

Provider order:

1. Gemini `gemini-2.5-flash`
2. Groq `llama-3.3-70b-versatile`
3. OpenRouter `deepseek/deepseek-chat`

## Local Setup

### Backend

```powershell
cd backend
python -m venv venv
.\venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --reload
```

### Frontend

```powershell
cd frontend
npm install
npm run dev
```

Frontend default local URL: `http://localhost:5173`  
Backend default local URL: `http://127.0.0.1:8000`

## Environment

Copy `.env.example` into your real environment setup and fill in the keys you need.

Important assistant variables:

- `GEMINI_API_KEY`
- `GROQ_API_KEY`
- `OPENROUTER_API_KEY`
- `GEMINI_CHAT_MODEL`
- `GROQ_CHAT_MODEL`
- `OPENROUTER_CHAT_MODEL`
- `LECTURE_ASSISTANT_SYSTEM_PROMPT`

Important security variables:

- `APP_SECRET`: set a long random value in production; rotating it invalidates existing sessions.
- `SESSION_COOKIE_SECURE=true`
- `SESSION_COOKIE_SAMESITE=none` for the deployed Render frontend/backend origins.
- `SESSION_JWT_ISSUER=mabaso-ai`
- `SESSION_JWT_AUDIENCE=mabaso-ai-web`

Production persistence variables:

- `SQLITE_DB_PATH=/data/mabaso_ai.db`
- `REQUIRE_PERSISTENT_DB=true`
- Mount a Render persistent disk at `/data` for the backend service.

The backend refuses unsafe temporary SQLite storage on Render when `REQUIRE_PERSISTENT_DB=true`. This keeps users, subscriptions, usage attempts, payment records, and saved materials tied to the account after restarts and across devices.

## Notes

- Account materials and billing usage are stored by signed-in email on the backend; browser storage is only a cache.
- PayFast checkout uses `PAYFAST_MERCHANT_ID` and `PAYFAST_MERCHANT_KEY` on the backend; paid users activate automatically after the PayFast webhook confirms the matching plan payment. Manual PayShap subscriptions still require `PAYSHAP_ACCOUNT_NAME` and `PAYSHAP_NUMBER`; admins verify those references from Admin Dashboard -> Payments before subscriptions activate.
- Voice input uses the browser speech recognition API.
- Optional voice reply playback uses browser text-to-speech.
- Admin dashboard range controls support `1 day`, `7 days`, `1 month`, and `1 year`.
- If Supabase Security Advisor reports `rls_disabled_in_public` or `sensitive_columns_exposed`, run `backend/sql/supabase_security_hardening.sql` in the affected Supabase project's SQL editor. The frontend does not use Supabase directly, so public `anon` and `authenticated` table access should stay blocked while backend service-role access continues.
